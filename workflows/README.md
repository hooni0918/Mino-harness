# 워크플로우

Claude Code의 `Workflow` 툴로 돌리는 오케스트레이션 스크립트입니다. 스킬·에이전트가 "무엇을·어떻게"라면, 여기 스크립트는 그것들을 **어떤 순서로, 어떤 게이트를 걸어 엮을지**를 결정론적으로(반복·조건·팬아웃) 담습니다. 둘 다 커밋을 만들지 않습니다.

## figma-to-pr.js — 입구 하네스

Figma URL 하나를 받아 화면 분류부터 QA 판정까지 흘려보내는 진입점입니다.

```
Workflow({ scriptPath: "workflows/figma-to-pr.js", args: "https://figma.com/..." })
```

### 동기

LLM 작업의 비용은 대부분 모델 등급에서 갈립니다. 전부 똑똑한 모델로 돌리면 정확하지만 비싸고, 전부 가볍게 돌리면 싸지만 엉성합니다. 그래서 **판단이 필요한 곳에만 비싼 모델을 씁니다** — 맨 앞에서 Opus로 한 번 분류하고, 실행은 분류기가 배정한 모델로 내려갑니다.

### 동작

먼저 Opus가 Figma를 직접 읽어 화면별 `changeType`(new/modify/qa-only)과 단계별 모델을 정합니다. 이후 각 화면이 파이프라인을 독립적으로 흐릅니다 — 수정(modify만) → 접근성 → 테스트 → 빌드 → QA. 각 단계는 [`.claude/agents/`](../.claude/agents/)의 전용 에이전트가 맡습니다.

단계마다 게이트가 걸려 있고, 게이트에서 떨어진 화면은 **조용히 사라지지 않습니다.** `gated()` 헬퍼가 실패를 `dropped[]`에 화면·단계·사유로 남깁니다(파이프라인의 기본 동작은 실패 항목을 `null`로 지우는 것이라, 사유가 사라지지 않게 마커로 통과시킵니다). 신규 화면은 파이프라인 밖으로 빠져 `guidance[]`("대화형 `/ios-workflow`로 진행하라")로 분리됩니다.

### 경계

- **커밋하지 않습니다.** 수정·식별자 부여는 워킹트리에 남고, 사람이 리포트와 diff로 검토한 뒤 커밋합니다.
- **신규 화면은 만들지 않습니다.** 계획 승인·리뷰 같은 사람 게이트가 필요한 작업이라 안내만 냅니다.
- Figma MCP(claude.ai 인증)에 의존하며, 백그라운드 실행에서 도구가 빠질 수 있습니다. AXe CLI는 QA 단계에서 필요합니다.
- 반드시 이 레포를 cwd로 두고 실행해야 합니다 — `.claude/agents/`의 에이전트가 등록돼 있어야 `agentType` 디스패치가 동작합니다.

## adversarial-harden.js — 자기개선 루프

이 번들 자신(에이전트·스킬·README)의 빈틈을 적대적으로 공격해 단단하게 만드는 워크플로우입니다.

```
Workflow({ scriptPath: "workflows/adversarial-harden.js" })
```

### 동기

AI가 쓴 프롬프트와 에이전트 정의에도 빈틈이 있습니다. 혼자 검토하면 자기 논리의 구멍을 못 봅니다. 그래서 여러 비평가가 차원을 나눠 동시에 공격하게 합니다.

### 동작

한 라운드는 두 단계입니다. 먼저 **차원별 비평가**(harness-truth / mino-arch-fit / a11y-coverage / test-quality)가 병렬로 산출물을 공격합니다 — 한 명이 다 보는 것보다 빈틈이 적습니다. 이어 각 발견을 **독립 검증자 3명이 "반증"을 기본값으로 재검**하고, 3표 중 2표 이상 살아남은 것만 `confirmed`로 채택합니다. 새 발견이 없는 라운드가 2회 연속이면 종료합니다(loop-until-dry).

### 경계

- **수정을 자동 반영하지 않습니다.** 검증을 통과한 결함도 오판일 수 있어, "확정 결함 목록"까지만 내고 반영은 사람이 diff로 판단합니다.
- 여기서는 판단이 필요한 **의미론적 결함**(문서가 약속한 게이트·흐름과 실제 코드의 괴리, 아키텍처 위반 등)만 다룹니다. grep으로 끝나는 결정론적 정합(호명 오류·죽은 링크·axe 명령 존재·금지 문자열)은 [`../scripts/check_consistency.py`](../scripts/check_consistency.py)가 CI에서 담당합니다.

> 병렬 적대 루프의 배경은 [`../docs/adversarial-improvement.md`](../docs/adversarial-improvement.md)를 참고하세요.
