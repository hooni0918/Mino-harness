# Mino QA 파이프라인

SwiftUI 피쳐가 끝났을 때 접근성 부여 → 테스트 작성 → 빌드 → 시뮬레이터 실행 → 판정을 순서대로 돌리는 QA 스킬입니다. 별도 UI 테스트 타깃을 만들지 않고 AXe로 화면을 밖에서 직접 눌러 봅니다.

## 동기

화면을 다 만들고 나면 반복 노동이 남습니다. 접근성을 붙이고, 테스트를 쓰고, 시뮬레이터에서 직접 눌러보고, 결과를 읽는 일. 매 피쳐마다 똑같이 반복되는데 사람이 매번 손으로 합니다.

이 스킬은 그 노동을 정해진 순서로 묶어 위임한 것입니다. "이 화면은 이렇게 QA한다"를 한 번 정해 프롬프트로 박아 두면, 다음 피쳐에서 같은 결정을 다시 내리지 않아도 됩니다.

## 파이프라인

다섯 단계가 순서대로 흐르고, 각 단계는 독립된 에이전트가 맡습니다. 앞 단계의 산출물이 뒤 단계의 입력이 됩니다.

- **① 접근성** (`accessibility-auditor`) — 뷰에 `accessibilityIdentifier`를 붙이고 식별자 매니페스트를 남깁니다. 이 식별자가 있어야 뒤 단계가 요소를 가리킬 수 있습니다.
- **② 테스트** (`test-author`) — 단위테스트(Swift Testing)와 AXe UI 시나리오를 작성합니다.
- **③ 빌드** (`build-runner`) — 빌드하고 시뮬레이터에 설치·실행합니다.
- **④ 실행** (`simulator-qa`) — AXe로 시뮬레이터를 조작하며 단계별 스크린샷을 남깁니다.
- **⑤ 판정** (`qa-reviewer`) — 스크린샷과 테스트 결과를 보고 PR용 QA 리포트를 냅니다.

각 단계는 별도 컨텍스트에서 돌고, 메인은 산출물 요약만 쥡니다. `--unit-only`(단위테스트까지만) / `--ui-only`(시나리오·실행만) 플래그로 범위를 좁힐 수 있습니다.

## 게이트

단계마다 통과 조건이 있어, 뒤 단계가 헛돌지 않게 앞에서 막습니다.

- 접근성 식별자가 하나도 안 붙으면 → 멈춤 (뒤 단계가 요소를 못 찾음).
- 단위테스트가 컴파일조차 안 되면 → 멈춤.
- 빌드 실패·앱 타깃 없음 → 멈춤.
- 빌드는 됐는데 시뮬레이터 설치·실행까지 못 가면(미가용 등) → 멈추지 않고 다음 단계를 **HOLD**로 넘깁니다(소프트 게이트).

## 한계 (반드시 인지)

- **한글 입력 불가** — AXe `type`은 US 키보드 문자만 지원합니다. 한글 흐름은 시드 데이터로 우회하거나 시나리오에서 뺍니다.
- **접근성 트리 의존** — 식별자 없는 화면은 선택자로 못 찾습니다. ①이 선행되지 않으면 ④는 무력합니다.
- **fire-and-forget** — AXe HID 명령은 전달만 보장하고 앱이 처리했는지는 모릅니다. 그래서 결과는 항상 스크린샷·`describe-ui`로 따로 검증하고, 모호하면 `qa-reviewer`가 "판정 보류"로 분리합니다.
- **게이트는 개수만 본다** — 접근성 게이트는 식별자 총개수가 0보다 큰지만 확인합니다. 요소별 완전 커버리지는 검사하지 않습니다.
