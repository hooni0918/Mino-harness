# Mino Router

들어온 작업을 화면 단위로 "무슨 작업인지" 분류하고, 어느 경로·어느 모델로 보낼지 정하는 스킬입니다. 분류와 라우팅만 하고, 실제 수정·테스트는 하지 않습니다. 신규 화면도 여기서 만들지 않습니다.

## 동기

작업을 시작할 때마다 같은 판단을 반복하게 됩니다. "이건 새 화면인가 기존 수정인가", "검증만 하면 되나", "이 정도 작업에 제일 비싼 모델을 써야 하나". 매번 사람이 정하면 느리고, 그렇다고 모든 단계를 가장 똑똑한 모델로 돌리면 비쌉니다.

그래서 이 판단을 앞단에 한 번 모았습니다. 비싼 모델로 딱 한 번 분류하고, 나머지 실행은 작업 복잡도에 맞는 모델로 내려보냅니다. **첫 판단만 무겁게, 실행은 가볍게.**

## 분류 방식

라우팅의 제1 기준은 **사람 게이트가 필요한가**입니다. 신규 화면 구현은 계획 승인·리뷰 같은 사람의 판단이 필요한 작업이라 배경에서 무인으로 돌리지 않고 대화형 경로로 보냅니다. 반대로 기존 화면 수정이나 QA는 기계 게이트(Figma 원본 대조·컴파일·접근성 식별자)가 사람 리뷰를 대신할 수 있어 무인 파이프라인으로 보냅니다.

분류는 추측이 아니라 Figma를 직접 읽고 내립니다. 그렇게 각 화면을 셋 중 하나로 판정합니다.

- **new** — 코드베이스에 없는 화면. 파이프라인에서 빼고, "대화형 `/ios-workflow`로 진행하라"는 안내만 냅니다.
- **modify** — 기존 화면에 디자인 변경 반영. Figma 원본과 대조하며 수정 → 접근성 → 테스트 → 빌드 → QA로 흐릅니다.
- **qa-only** — 코드 변경 없이 검증만. 수정 단계를 건너뛰고 접근성부터 QA까지 돕니다.

함께 화면의 복잡도(low/medium/high)와 한 줄 요약도 매깁니다. 복잡도는 다음 단계의 모델을 고르는 근거가 됩니다.

## 모델 배정

분류 한 번만 Opus로 판단하고, 실행 단계는 기본 Sonnet으로 내려갑니다. 정말 기계적인 단계는 더 가볍게 떨어뜨릴 수 있지만, 테스트·분석·리뷰가 섞인 단계는 Sonnet 아래로 내리지 않습니다 — 실제로 Haiku로 돌려 보니 틀린 지적이 많아 격상했습니다. 모델이 코드에 고정돼 있지 않고 **분류기가 복잡도를 보고 매번 정한다**는 것이 핵심입니다.

## 경계

- 실제 실행은 이 스킬이 아니라 [`figma-to-pr.js`](../../../workflows/figma-to-pr.js)가 합니다. 이 스킬은 그 워크플로우가 따르는 **분류 기준과 라우팅 규칙**만 정의합니다.
- 신규 화면은 만들지 않습니다. 안내(`guidance`)를 내는 데서 멈춥니다.
- 게이트에서 떨어진 화면은 조용히 사라지지 않고, 최종 리포트의 `dropped[]`에 화면·단계·사유로 남습니다.
- 분류는 Figma를 읽어야 하므로 Figma MCP에 의존합니다. 백그라운드 실행에서 도구가 빠지면 좌표·추측으로 우회하지 않고 멈춰서 보고합니다.
