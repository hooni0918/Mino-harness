# 에이전트

하네스에서 실제로 손을 움직이는 일꾼들입니다. 각 에이전트는 **독립된 컨텍스트**에서 돌고, 자기 일에 필요한 도구만 쥐며(예: 판정 에이전트는 편집 권한이 없음), 작업 기준이 필요하면 전문 [스킬](../skills/)을 소환합니다. 전부 `model: sonnet`으로 돕니다.

공통 원칙이 둘 있습니다.

- **커밋하지 않습니다.** 수정·산출물은 워킹트리와 리포트에 남기고, 반영은 사람이 diff로 판단합니다.
- **역할을 섞지 않습니다.** 코드를 고치는 에이전트와 결과를 판정하는 에이전트를 나눕니다. 만드는 쪽이 자기 결과를 스스로 통과시키지 않게 하기 위해서입니다.

## 수정 경로

### [screen-modifier](screen-modifier.md)

기존 SwiftUI 화면을 Figma 디자인 변경에 맞춰 수정합니다. 신규 화면 생성은 하지 않습니다(그건 대화형 `/ios-workflow`의 일). 핵심은 **수렴 루프**입니다 — 수정한 뒤 Figma 원본을 다시 fetch해 재대조하고, 차이 0건까지 반복합니다. 자기가 만든 차이 목록이 아니라 항상 Figma 원본을 기준으로 재검합니다(놓친 항목을 자기 목록으로는 못 잡으므로). 5라운드 안에 수렴 못 하면 `converged=false`와 남은 차이를 그대로 보고합니다. 마크업만 손대고 로직·하위 레이어는 건드리지 않습니다.

## QA 파이프라인 (순서대로)

### [accessibility-auditor](accessibility-auditor.md)

파이프라인 1단계. SwiftUI 뷰에 `accessibilityIdentifier`를 붙입니다. 이 식별자는 VoiceOver 접근성이자, 뒤 단계가 요소를 찾는 선택자입니다. 이름은 표시 텍스트가 아니라 **역할**에서 옵니다(`"로그인"`이 아니라 `Login.submitButton`) — 지역화·문구 변경에 안 깨지게. 반복되는 행은 안정적 ID로, 상태를 가진 컨트롤은 상태 확인 수단까지 함께 부여합니다. 결과로 식별자 매니페스트(`qa/manifests/<Screen>.json`)를 남겨 파이프라인이 끊겨도 이 지점부터 다시 잇게 합니다.

### [test-author](test-author.md)

파이프라인 2단계. 두 종류를 만듭니다 — 순수 로직 단위테스트(Swift Testing)와, 화면 흐름 UI 시나리오(AXe batch 파일). 단위테스트 기대값은 프로덕션과 같은 식으로 도출하지 않고(항상 통과하는 자가검증이 되므로) 독립 계산한 고정값을 씁니다. UI 시나리오는 accessibility-auditor가 넘긴 매니페스트의 식별자를 `--id` 선택자로 엮어 짭니다. 한글 입력 단계는 AXe 제약상 시나리오에서 빼거나 시드 데이터로 우회합니다.

### [build-runner](build-runner.md)

파이프라인 3단계(빌드 게이트). 빌드해서 시뮬레이터에 설치·실행합니다. 빌드 명령의 1차 출처는 프로젝트 `CLAUDE.md`입니다. **앱 타깃이 없는 저장소(이 QA 번들 자체가 그렇습니다)에서는 빌드를 시도하지 않고 "대상 없음"을 보고합니다** — 에러가 아니라 정상적인 조기 종료. 빌드가 실패하면 첫 에러와 `파일:라인`만 추려 보고하고 **직접 고치지 않습니다**. 딥링크 규약이 있으면 QA 대상 화면으로 바로 진입시키고, 없으면 추측으로 내비게이션하지 않습니다.

### [simulator-qa](simulator-qa.md)

파이프라인 4단계. AXe로 부팅된 시뮬레이터에서 시나리오를 실제 실행하고 단계별 스크린샷을 증거로 남깁니다. 실행 전에 `describe-ui`로 접근성 트리를 덤프해 시나리오의 식별자가 실제 존재하는지 먼저 대조합니다 — 없으면 좌표로 우회하지 않고 accessibility-auditor 재실행을 제안합니다. 실행·관측 전용이라 코드 수정도, 정오 판정도 하지 않습니다.

### [qa-reviewer](qa-reviewer.md)

파이프라인 5단계(마지막). 스크린샷·로그·테스트 결과를 사람 대신 1차로 읽고 `PASS`/`FAIL`/`PARTIAL`/`HOLD`로 판정합니다. 어긋난 첫 단계를 짚어 원인을 한 줄로 적고, 스크린샷만으로 확정 불가한 항목은 추측으로 통과시키지 않고 "판정 보류"로 분리합니다. 이 리포트가 PR 본문에 붙는 QA 요약이 됩니다.
